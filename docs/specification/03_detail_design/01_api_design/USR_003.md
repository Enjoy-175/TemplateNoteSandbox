# USR_003 ユーザー削除API
- Method: DELETE
- Path: /api/users/{userId}
- 認証: 不要（冪等）

## Input
- userId（UUID, 必須, UUID形式, 例: "5b8e7e4a-4f21-4d0e-b72f-8d5a2d5c9f3b"）

## Validation Rule
- Rule V001: userId 必須チェック
  - 対象: userId
  - 条件: null
  - 判定: NG
  - エラーコード: E-400-VALIDATION
  - メッセージ: "入力値が不正です。"

- Rule V002: userId 形式チェック
  - 対象: userId
  - 条件: UUID 形式でない
  - 判定: NG
  - エラーコード: E-400-VALIDATION
  - メッセージ: "入力値が不正です。"

### チェック順序
1. V001 → NGなら終了
2. V002 → NGなら終了
3. 全てOKなら次の処理へ

## Process
- Step 1: 入力バリデーション
  - Validation Rule を順に適用
  - NG: E-400-VALIDATION を返して終了

- Step 2: ユーザー存在確認
  - userId でユーザーを検索
  - 条件: ユーザーが存在しない場合でもエラーにしない
  - NG: なし（存在しない場合は Step 3 をスキップ）

- Step 3: ユーザー削除
  - users テーブルから削除
  - トランザクション開始
  - 失敗: ロールバックし E-500-DB

- Step 4: レスポンス生成
  - No Content を返却

## Output
### パターン1: 成功
- 型: なし
- HTTP: 204

### パターン2: Validation Error（例: V001）
- 型: JSON
- HTTP: 400
- 例:
  {
    "code": "E-400-VALIDATION",
    "message": "入力値が不正です。",
    "details": [
      { "field": "userId", "message": "入力値が不正です。" }
    ],
    "operation": "delete"
  }

### パターン3: システムエラー
- 型: JSON
- HTTP: 500
- 例:
  {
    "code": "E-500-DB",
    "message": "システムエラーが発生しました。",
    "details": null,
    "operation": "delete"
  }

## Error 定義
- E-400-VALIDATION
  - 発生条件: userId の必須/形式チェックNG
  - Input例: userId = "invalid-uuid"
  - 検出Step: Step 1
  - Output例: Output パターン2

- E-500-DB
  - 発生条件: DBアクセス例外
  - 検出Step: Step 3
  - Output例: Output パターン3

- E-500-UNEXPECTED
  - 発生条件: 予期しない例外
  - 検出Step: 全Step
  - Output例:
    {
      "code": "E-500-UNEXPECTED",
      "message": "予期しないエラーが発生しました。",
      "details": null,
      "operation": "delete"
    }
